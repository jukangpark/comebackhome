## 0. 큰 그림: 3D 모델은 3가지 레이어로 되어 있다

3D 캐릭터 하나는 사실 세 겹의 정보 묶음이야.

```
┌─────────────────────────────────────────┐
│  3) 리깅 + 애니메이션 (움직임)            │  ← 뼈대와 동작
├─────────────────────────────────────────┤
│  2) 텍스처 / 머티리얼 (겉면 색·질감)      │  ← 표면에 입히는 그림
├─────────────────────────────────────────┤
│  1) 메시 (형태 = 폴리곤 덩어리)           │  ← 뼈대 없는 조각상
└─────────────────────────────────────────┘
```

Meshy image-to-3D가 주는 건 기본적으로 **1) + 2)** 까지.
**3) 리깅/애니메이션은 별도 작업**이야. (그래서 이전 문서에서 계속 강조한 것.)

## 1. 메시(Mesh) — 형태

### 정점 / 엣지 / 폴리곤
3D 형태는 결국 **점들을 이어 만든 삼각형(또는 사각형)의 집합**이야.

- **Vertex(정점)** — 3D 공간의 점 하나 `(x, y, z)`
- **Edge(엣지)** — 정점 두 개를 잇는 선
- **Face / Polygon(면)** — 엣지로 둘러싸인 면. 보통 삼각형(tri) 또는 사각형(quad)
- **Mesh(메시)** — 이 면들이 모여 만든 껍데기 전체

> 💡 컴퓨터 그래픽은 결국 전부 **삼각형**으로 그린다. 사각형도 렌더링 직전엔 삼각형 2개로 쪼개짐.

### Polygon count (폴리곤 수) = 성능의 핵심 지표
- 폴리곤이 많을수록 → 디테일↑, **용량↑, 렌더링 무거움↓(느려짐)**
- 웹앱(실시간 렌더)에서는 **가벼움이 생명**. 영화 CG는 수백만 폴리곤 써도 되지만, 브라우저에서 도는 아바타는 **3만(30k) 정도면 충분.**
- Meshy `target_polycount` 파라미터가 이걸 조절함.

### Topology (토폴로지) — 폴리곤이 "어떻게 배열됐나"
같은 형태라도 폴리곤을 어떻게 까느냐가 다름.

- **Triangle(삼각형) 토폴로지** — 실시간 렌더에 가볍고 표준. 웹앱 아바타는 이거.
- **Quad(사각형) 토폴로지** — 애니메이션/변형에 유리(관절이 깔끔하게 접힘). **리깅할 거면 quad가 좋음.**
- **"깨끗한 토폴로지"** = 관절 부위에 엣지가 흐름(edge loop) 따라 잘 배열된 것. 자동생성 3D는 토폴로지가 지저분한 경우가 많아서 리깅이 어려워지는 원인이 되기도 함.

> 우리 프로젝트 판단: **정적 아바타면 triangle, 나중에 리깅하려면 quad.**

## 2. 텍스처 / 머티리얼 — 겉면

메시는 회색 찰흙 덩어리일 뿐. 여기에 **색과 질감을 입히는 게 텍스처.**

### UV 매핑
- 3D 표면을 2D 평면으로 펼친 "전개도"가 **UV**.
- 종이 인형 만들 때 도면 펼치듯, 3D 표면을 평평하게 펴서 그 위에 그림(텍스처)을 그림.
- 그 2D 그림을 다시 3D에 감싸면 → 강아지 털 무늬가 몸에 입혀짐.

### 텍스처의 종류 (PBR)
현대 3D는 표면을 여러 장의 "맵(map)"으로 표현해 — 이걸 **PBR(물리 기반 렌더링)** 이라 함.

| 맵 | 역할 |
|---|---|
| **Base Color / Albedo** | 순수한 색 (털 색, 무늬) — 필수 |
| **Normal** | 표면의 미세 요철 (털 결, 주름)을 가짜로 표현 |
| **Metallic** | 금속성 여부 (동물엔 거의 0) |
| **Roughness** | 거칠기/광택 (코는 반짝, 털은 매트) |
| **Emission** | 스스로 빛나는 부분 (동물엔 보통 없음) |

- Meshy `enable_pbr: true` 하면 이 맵들을 다 만들어줌 → 사실적이지만 **용량↑**.
- **웹 아바타 MVP는 Base Color만으로도 충분**. `enable_pbr: false` 권장.

### 머티리얼(Material)
텍스처 맵들 + 렌더링 설정(투명도, 광택 등)을 묶은 것. three.js에서 `MeshStandardMaterial` 같은 걸로 다룸.

## 3. 리깅 & 애니메이션 — 움직임 (제일 어려운 부분)

### 리깅(Rigging) = 뼈대 심기
메시는 관절 없는 마네킹. 움직이게 하려면 안에 **골격을 넣어야** 함.

- **Bone / Joint(뼈/관절)** — 움직임의 단위. 뼈를 회전시키면 주변 살이 따라 움직임.
- **Skeleton / Armature(골격)** — 뼈들을 부모-자식으로 연결한 계층 구조.
  (예: 어깨뼈를 돌리면 팔·손이 다 따라옴)
- **Skinning / Weight Painting(스키닝/웨이트)** — "이 뼈가 움직이면 겉의 어느 정점이 **얼마나** 따라올지"를 칠하는 작업. 리깅에서 제일 손이 많이 감.
  (예: 팔꿈치를 접을 때 위팔 살은 100% 따라오고, 팔뚝 살은 자연스럽게 접히도록 비율 조정)

```
       [척추]
        │
   ┌────┼────┐
 [앞다리]  [뒷다리]      ← 이런 계층이 "skeleton"
        │
      [꼬리]

각 뼈 ──웨이트──▶ 주변 메시 정점들
```

### 인간 리깅 vs 동물 리깅 — 왜 차이가 나나
- **인간형(humanoid)**: 골격 구조가 표준화됨(척추 + 팔 2 + 다리 2 + 머리). 그래서:
  - **Meshy 자동 리깅 API 지원** (단, humanoid 전용)
  - **Adobe Mixamo** 로 자동 리깅 + 무료 모션 라이브러리 사용 가능
  - → 사람 캐릭터는 "업로드하면 자동으로 걷고 춤춤"
- **동물(4족 quadruped)**: 표준 골격이 없음(다리 4개, 꼬리, 주둥이, 귀). 그래서:
  - Meshy/Mixamo **자동 리깅 안 됨** ❌
  - 대부분 **수동 리깅**(Blender에서 뼈 하나하나) 또는 동물 전용 리그 툴 필요
  - → **이게 우리 프로젝트에서 애니메이션이 까다로운 핵심 이유**

### 애니메이션(Animation)
리깅이 끝나야 가능. 뼈들의 시간에 따른 회전값을 기록한 게 애니메이션 클립.

- **Keyframe(키프레임)** — 특정 시점의 포즈. 사이는 컴퓨터가 보간(자동으로 부드럽게 연결).
- **Idle / Wag / Sit 같은 클립** 을 만들어 상황별로 재생.
- three.js에서는 `AnimationMixer` 로 glb 안의 애니메이션 클립을 재생함.

> 🐕 **우리 프로젝트 현실적 전략:**
> MVP는 리깅 없이 **정적 메시 + 카메라 회전 + 살짝 위아래 bounce(코드로 통째 이동)** 로 시작.
> "뼈로 꼬리 흔들기" 같은 건 동물 리깅 R&D를 따로 판 뒤 2차로.

## 4. 파일 포맷 — `.glb`가 왜 웹 표준인가

| 포맷 | 설명 | 웹 적합성 |
|---|---|---|
| **glTF (.gltf)** | "3D계의 JPEG". 웹 표준. 텍스트(json) + 별도 파일들 | ✅ |
| **GLB (.glb)** | glTF를 **바이너리 1개 파일로 묶은 것** (메시+텍스처+애니 전부 포함) | ✅✅ 최적 |
| FBX | 게임/애니 업계 표준. 리깅·애니 잘 담음. 웹은 무거움 | △ |
| OBJ | 아주 단순(메시+색만). 애니메이션 못 담음 | △ |
| USDZ | Apple AR용 (iOS 3D 뷰어) | iOS AR용 |
| STL | 3D 프린팅용. 색·텍스처 없음 | ❌ |

> **결론: 웹앱은 GLB 하나면 끝.** 메시·텍스처·(있다면)애니메이션까지 한 파일에 다 들어있어서
> three.js `GLTFLoader` 로 바로 로드됨. Meshy `target_formats: ["glb"]` 로 요청.

## 5. 브라우저에서 3D 띄우기 — 우리 스택 매핑

| 개념 | 도구 | 역할 |
|---|---|---|
| **three.js** | WebGL 저수준 3D 라이브러리 | 브라우저에서 3D 렌더링의 근간 |
| **React Three Fiber (R3F)** | three.js를 React 컴포넌트로 | `<mesh>`, `<Canvas>` 처럼 JSX로 3D 작성 |
| **@react-three/drei** | R3F 헬퍼 모음 | `<OrbitControls>`(마우스 회전), `useGLTF`(glb 로드) 등 |

가장 단순한 예시(개념만):
```tsx
import { Canvas } from "@react-three/fiber";
import { OrbitControls, useGLTF } from "@react-three/drei";

function PetModel({ url }: { url: string }) {
  const { scene } = useGLTF(url);   // .glb 로드
  return <primitive object={scene} />;
}

export function PetViewer({ glbUrl }: { glbUrl: string }) {
  return (
    <Canvas camera={{ position: [0, 1, 3] }}>
      <ambientLight intensity={0.8} />
      <directionalLight position={[5, 5, 5]} />
      <PetModel url={glbUrl} />
      <OrbitControls />           {/* 마우스로 빙글빙글 */}
    </Canvas>
  );
}
```
이 정도면 "회전 가능한 정적 강아지 아바타"는 완성. 여기서 조명·배경·bounce 애니메이션을 붙여나가면 됨.

## 6. 용어 빠른 사전 (치트시트)

| 용어 | 한 줄 정의 |
|---|---|
| Mesh | 폴리곤으로 만든 3D 형태 (뼈대 없는 조각상) |
| Vertex/Edge/Face | 정점 / 선 / 면 — 메시의 구성 요소 |
| Polygon count | 면 개수. 많을수록 정밀하지만 무거움 |
| Topology | 폴리곤 배열 방식 (tri vs quad) |
| UV | 3D 표면을 2D로 펼친 전개도 (텍스처 붙일 좌표) |
| Texture | 표면에 입히는 2D 이미지 (색·질감) |
| PBR | 여러 맵으로 사실적 표면 표현하는 방식 |
| Material | 텍스처 + 렌더 설정 묶음 |
| Rigging | 메시 안에 뼈대(골격) 심기 |
| Skeleton/Bone | 골격 / 개별 뼈 |
| Skinning/Weight | 뼈 움직임에 살이 얼마나 따라올지 설정 |
| Animation/Keyframe | 뼈의 시간별 동작 / 특정 시점 포즈 |
| glTF/GLB | 웹 표준 3D 포맷 (GLB=바이너리 단일 파일) |
| three.js / R3F | 브라우저 3D 렌더 라이브러리 / 그 React 버전 |

## 7. 우리 프로젝트에 적용하는 결론

1. **Meshy가 주는 것**: 메시 + 텍스처(1+2 레이어)까지. 이걸 GLB로 받아 R3F로 렌더 → **정적 아바타 MVP 즉시 가능.**
2. **리깅/애니메이션(3 레이어)은 동물이라 자동화 안 됨** → MVP에서 제외, 별도 R&D.
3. **파라미터 기본값**: `topology: triangle`, `polycount: 30k`, `should_texture: true`, `enable_pbr: false`, `formats: [glb]`.
4. **다음 공부 주제**: 동물 리깅 방법(Blender 수동 리깅 / 동물 전용 오토리거 조사), three.js 조명·환경맵으로 예쁘게 보이게 하기.
